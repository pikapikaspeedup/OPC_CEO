# CEO 备忘录：Obsidian 上下文可控化与 Scheduler 前台入口缺口

日期：2026-03-29

## 一页结论

当前系统已经具备两类能力，但用户感知与产品入口不匹配：

1. Obsidian 聊天上下文能力存在，但过去的默认行为过于隐式，导致用户误以为系统在“偷偷塞上下文”。
2. Scheduler 定时任务能力在后端和通用前端里已经存在，但它没有进入 CEO 主视角，也没有形成“可派发、可追踪、可回看”的经营闭环。

结论不是“能力不存在”，而是“默认交互与入口层级设计错误”。

---

## 一、Obsidian 聊天上下文的场景缺陷

### 1. 用户感受到的缺陷

在 Obsidian 插件里与 Antigravity 对话时，用户会遇到三类明显问题：

1. 主动拉入的文件，发送后在消息气泡里消失。
2. Related Notes / Knowledge Context 会在发送时临时注入，但发送前看不完整、发送后也无法审计。
3. 隐式注入的上下文无法逐项剔除，用户对“这条消息到底带了什么”没有控制权。

### 2. 根因判断

这不是单点 bug，而是三层实现叠加造成的体验错位：

1. 发送前使用本地乐观渲染，临时显示完整 prompt。
2. 服务端正式保存时，会把文件引用拆成结构化 file item，而不是继续保留在纯文本里。
3. 插件旧版正式渲染器只显示 text item，不显示 file item；同时 Knowledge Context 是发送时临时拼接，所以不会自然进入“可审计上下文条”。

### 3. 产品层面的缺陷定义

这类问题的本质不是“渲染小 bug”，而是权限与可见性模型不一致：

1. 注入行为是系统主动做的，但知情权没有给用户。
2. 注入结果会影响模型回答，但发送前不可验证。
3. 用户不能在消息级别决定是否带入 Related Notes。

对于知识工作产品，这会直接削弱信任。

### 4. 已落地方案

当前建议并已按这个方向收口：

1. Smart Context 和 Related Notes Context 默认改为 Manual，而不是 Automatic。
2. 只有用户主动点击 Smart 或 Related，本条消息才带入对应上下文。
3. 发送前的上下文条必须显示将要带入的项目，并支持逐项剔除。
4. 正式消息渲染补齐 file item，避免“发送前看得到，发送后消失”。

### 5. 产品原则

后续应坚持一个简单原则：

默认不要替用户加知识上下文。

上下文注入只能有两种方式：

1. 用户显式点击。
2. 用户在设置里明确开启自动模式。

## 二、Scheduler 定时任务的现状判断

### 1. 当前能力不是空白

项目已经具备完整的 Scheduler 基础能力：

1. 有 job 数据结构，支持 cron / interval / once。
2. 有 job 的创建、编辑、删除、手动触发 API。
3. 有定时轮询执行器。
4. 有审计事件写入。
5. 支持 OPC 风格的 create_project 定时派发。

换句话说，后端能力已经不是概念验证，而是可以工作的原型。

### 2. 运行事实

Scheduler 会在服务端启动时初始化，因此并非“只能手动 trigger 的假功能”。

但它目前更像“Ops 面板里的系统工具”，还不是“CEO 工作台的一等入口”。

### 3. 为什么用户会感觉前端没有入口

因为从 CEO 的主工作流来看，Scheduler 的入口几乎等于没有：

1. SchedulerPanel 只挂在侧栏的 Operations 分区里。
2. CEO Dashboard 没有任何 Scheduler 卡片、入口按钮或待办视图。
3. 项目主工作台没有“把当前项目转成定时任务”的入口。
4. CEO 事件流按钮目前没有真正执行动作。
5. 当前 Scheduler 表单更偏工程配置，不像 CEO 可直接操作的经营工具。

因此，虽然技术上前端“有一个面板”，产品上仍然可以定义为“CEO 侧没有入口”。

---

## 三、对 CEO 视角的真正缺口定义

当前缺的不是定时器本身，而是以下三层闭环：

### 1. 缺调度入口

CEO 在看日报、事件、部门负载时，无法顺手把一个动作设为每天、每周、或一次性任务。

### 2. 缺调度语义

现在的 Scheduler 更像技术参数表单，而不是业务动作：

1. 每天 9 点收取某部门日报。
2. 每周一巡检某项目健康度。
3. 明早为某部门自动创建一个 ad-hoc 项目。

这些 CEO 真正常用的动作，没有被产品化成模板。

### 3. 缺回看与追责

即使任务已定时创建，CEO 视角也看不到：

1. 这个部门有哪些 active scheduled jobs。
2. 哪些 job 最近失败了。
3. 哪些 job 实际创建了哪些 project。
4. 某次 daily digest 是人工触发还是定时触发。

---

## 四、建议方案

### P0：把 Scheduler 抬到 CEO 主视角

建议新增两个入口：

1. 在 CEO Dashboard 的命令区旁边增加 Schedule 入口。
2. 在 Department 卡片 / 项目卡片上增加 Schedule 动作。

目标不是替换现有 Operations 面板，而是把它从“深层工具”升级为“主流程入口”。

### P1：把复杂表单包装成 CEO 模板动作

不要让 CEO 直接填写底层字段，先提供 3 个模板：

1. 定时收日报。
2. 定时健康巡检。
3. 定时派发部门任务。

用户先选模板，再补最少参数。底层仍然写入现有 scheduler job 数据结构。

### P1：让调度结果进入 CEO 可见流

需要把 Scheduler 的状态显示到 CEO 主屏：

1. Upcoming jobs。
2. 最近执行结果。
3. 失败告警。
4. 由 job 自动创建的项目回链。

### P2：把事件流按钮做成真动作

CEO Dashboard 里的 event.actions 现在只有按钮外观，没有实际行为。应至少接通：

1. 打开项目。
2. 打开部门。
3. 创建对应调度任务。
4. 跳转到 Scheduler 编辑。

---

## 五、建议排期顺序

### 第一阶段

1. 在 CEO Dashboard 增加 Scheduler 入口。
2. 在 Dashboard 展示最近 5 个 jobs 与执行状态。
3. 补齐事件按钮跳转能力。

### 第二阶段

1. 加入 CEO 级调度模板。
2. 支持从 Department / Project 一键创建 job。
3. 在 Daily Digest 和审计事件中回链 scheduler job。

### 第三阶段

1. 加入部门级默认计划。
2. 加入失败重试与升级告警。
3. 加入“从 CEO 自然语言命令直接创建 scheduler job”的能力。

---

## 六、最终判断

当前系统在“定时任务能力是否存在”这个问题上，答案是：存在。

但在“CEO 是否真正能用它”这个问题上，答案是：还没有。

因此下一步不该再讨论要不要做 Scheduler，而是应该直接把它从 Operations 工具，提升成 CEO 工作台的一等公民。

# 快速项目可视化交互问题分析（2026-04-17）

## 范围

基于当前“快速项目 / Ad-hoc Project”详情页截图，对可视化表达与交互理解成本做静态评审。

说明：

- 本次结论基于单张页面截图
- 未包含 hover / focus / 点击态的真实运行验证
- 重点关注首屏理解、操作优先级、状态表达与信息结构

## 核心问题重定义

用户补充后，这个页面最核心的问题可以更准确地定义为：

- 不是“信息有点乱”
- 而是“结果不可检视，导致任务是否真正完成无法被验证”

也就是说，当前界面的主要缺陷不是视觉样式，而是缺失一条清晰的完成证据链。

一个用户在看到 `completed` 之后，本应立刻能回答 4 个问题：

1. 到底产出了什么
2. 产物在哪里
3. 完成依据是什么
4. 是否已经通过验证，而不是只是 run 结束

当前页面这 4 个问题都没有被直接回答。

## 10 个关键问题

### 1. 页面主任务不够聚焦，标题过长且缺乏层级裁剪

当前主标题直接展示：

- 部门名
- 项目名
- 完整任务句子

导致首屏视觉重心被一整串长文本占满，用户很难在 2 秒内看清：

- 这是什么项目
- 现在完成到哪一步
- 接下来该看哪里

问题本质不是“字多”，而是没有把：

- 项目容器
- 执行目标
- 当前结果

拆成不同层级。

### 2. “已完成”状态很醒目，但没有和下方内容形成因果闭环

右上角绿色 `已完成` 很明确，但下方 `Pipeline Stages 0/0 completed`、`Prompt Runs 1`、卡片摘要之间没有形成统一叙事。

用户会出现典型疑问：

- 为什么项目已完成，但 pipeline 是 `0/0`
- 这是没有流程，还是流程没显示出来
- 完成的是项目、run，还是某个产物

状态存在，但状态解释缺失。

进一步说，这里缺的不是状态颜色，而是“完成证明”：

- 没有完成标准
- 没有校验结果
- 没有结构化产物摘要
- 没有证据入口

### 3. “Pipeline / Operations / Deliverables” 是一级导航，但页面主体没有匹配的主内容密度

当前 Tab 看起来像强切换导航，但首屏真正被看到的还是：

- 部门能力卡片
- Prompt Runs

而不是 Pipeline 主体。

这会让用户以为自己进入的是“流程页”，实际看到的却像“项目概览 + run 摘要”，造成导航承诺和内容兑现不一致。

### 4. 左侧栏信息重复，造成强干扰而不是强导航

左侧同时出现：

- Current Project
- Project Tree
- 多个名字极其相似的项目卡片

而主区顶部又再次展示同一个项目标题。

结果是同一条项目语义被重复讲了 3 次，但用户真正需要的：

- 当前项目与其他项目的区别
- 当前选中的层级
- 上下项目之间的关系

并没有被更清晰地表达。

### 5. 视觉对比度偏低，内容可扫读性不足

页面整体是深色低对比风格，但很多二级信息使用了：

- 灰色小字
- 细线描边
- 低亮度标签

在大屏上看起来“高级”，实际会让用户快速扫读时漏掉关键信息，尤其是：

- 时间
- 标签类别
- 结构说明
- 卡片边界

这类本应支持快速判断的信息。

### 6. 关键操作区缺少主次分层，顶部图标语义不够自解释

右上角同时出现：

- `派发流水线`
- 编辑
- 复制/归档类图标
- 删除

但这些操作没有明显主次，也缺少清晰的文本辅助。

尤其在项目已经 `completed` 的情况下，用户不知道：

- 此时最合理的下一步是复用、重跑、查看交付物，还是派发流水线
- 红色删除是否危险且是否有确认
- 中间两个纯图标分别是什么

图标存在，但操作模型不透明。

### 7. “能力声明卡片”信息量大，但不支持决策动作

中间的大卡片展示了：

- 部门
- skills
- workflowRef
- skillRefs

这对于调试很有价值，但对普通项目浏览来说更像配置面板，不像执行面板。

它占据首屏大面积空间，却不能直接回答用户最关心的三个问题：

- 这次为什么命中这个 workflow
- 这次产出了什么
- 我现在下一步能做什么

信息是“底层实现导向”的，不是“任务结果导向”的。

### 8. Prompt Runs 卡片缺乏结果消费入口，摘要可见但不可操作

当前 Prompt Run 卡片里已经有：

- 成功状态
- 时间
- 耗时
- 结果摘要

但没有明显的：

- 查看完整结果
- 查看产物文件
- 展开结构化详情
- 重跑本次 run

入口。

结果被展示成一段长文字，而不是可消费的结果对象。

这正是“无法检视任务真正完成情况”的直接来源，因为用户看不到：

- 结果文件
- 核心指标
- 验证状态
- 失败或跳过项
- 本次 run 的输入与输出边界

### 9. 项目页把“系统结构”和“业务结果”混在一起，用户心智负担偏重

页面同时强调：

- Project
- Department
- Skill
- Workflow
- Pipeline
- Prompt Run

这些都是系统内部概念。

但用户真正想确认的业务事实通常是：

- 今天的大事件是否已经生成
- 生成了几条
- 去哪里看详情
- 能不能继续派发后续动作

当前界面把“系统怎么做的”放在“做出了什么”前面，导致理解路径倒置。

从结果可验证性的角度看，顺序应该反过来：

1. 先看业务结果
2. 再看验证结论
3. 最后才看 skill / workflow / pipeline 等实现结构

### 10. 空态与异常态语义不清，容易让用户误判系统是否有 bug

`Pipeline Stages 0/0 completed` 在视觉上像一个未正确加载的区域，而不是一个被设计过的“本项目无 pipeline，仅有 prompt run”的空态。

这类表达会引发两个误判：

1. 用户以为数据没回来
2. 用户以为项目结构异常

也就是说，当前不是“没有内容”，而是“空态没有被设计成一种明确状态”。

## 优先级建议

### P1：必须先改

1. 重构首屏信息层级，先展示项目结果与下一步，再展示 department / skill / workflow 细节
2. 修正 `completed` 与 `0/0 pipeline` 的状态叙事冲突，给出明确空态文案
3. 给 Prompt Runs 增加明确结果入口，而不是只给摘要文本
4. 重做顶部操作主次，保留一个主 CTA，其余操作显式命名

### P2：第二批优化

1. 降低左侧重复信息密度，强化“当前选中”和“相邻项目差异”
2. 提升二级文本、标签、边框的亮度和对比度
3. 将“能力声明”折叠成诊断区，默认不占据首屏核心

### P3：体系化优化

1. 区分“执行结构视图”和“业务结果视图”
2. 为 adhoc/prompt-only 项目建立专门版式，而不是复用 pipeline 项目框架

## 结论

当前页面的最大问题不是“丑”，而是：

- 首页没有把结果、状态、下一步组织成一个顺畅的阅读路径
- `completed` 不是“完成证据”，只是“系统状态标签”
- 用户无法直接检视任务产物，所以也无法判断任务是否真正完成

它更像一个把内部执行结构直接暴露出来的调试界面，而不是一个面向日常使用的项目结果页。

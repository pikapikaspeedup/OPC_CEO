# Antigravity 长期演进大节点路线图

日期：2026-04-07
范围：基于当前 Antigravity 代码现状、CEO/OPC 愿景、以及与 craft-agents 的对比研究，给出未来 6-18 个月的主线演进节点。

## 一句话判断

Antigravity 接下来不该继续朝“通用 coding shell”方向分散精力，而应把自己明确做成一个可运营的 AI 软件组织系统：

1. 下层先补硬执行内核。
2. 中层补治理闭环。
3. 上层再把 CEO、Department、OKR、Memory、Scheduler 做成真的“虚拟公司操作系统”。

## 北极星

Antigravity 的北极星不是“像另一个 IDE Agent 一样会聊天、会跑工具”，而是：

> 让一个由 CEO、部门、员工、规则、记忆、技能、计划和例行节奏组成的 AI 组织，能够围绕真实项目长期运转。

对应到产品对象：

| 现实公司对象 | Antigravity 对应对象 |
|:--|:--|
| 公司 CEO | CEO Office / CEO Agent |
| 部门 | Workspace + Department Config |
| 员工 | Workflow / Role / Stage Executor |
| 制度 | Template / Rules / Approval Policy |
| 知识沉淀 | Department Memory / Org Memory |
| 例会与汇报 | Scheduler / Digest / Approval / Intervention |
| 项目推进 | Project / Run / Stage Runtime |

## 战略边界

未来路线要坚持四条边界。

1. 不直接把 Antigravity 建立在 craft 之上。
2. 不优先做一个更像 IDE 的通用 agent shell。
3. 不在执行内核还没统一前，继续横向扩很多 provider 和炫 UI。
4. 不让 CEO 的写操作绕开治理和审计主链路。

## 大节点总览

| 节点 | 时间带 | 核心目标 | 成功标志 |
|:--|:--|:--|:--|
| 节点 1：执行底座收口 | 0-6 周 | 把 Stage Runtime 建立在统一事件化 backend 契约上 | 单 stage 可在不同 executor 上运行，Run 状态不再绑死 transport 细节 |
| 节点 2：治理内核闭环 | 4-10 周 | 审批、配额、人工 gate、干预、调度真正能驱动运行时 | CEO 的治理操作会改变 Project/Run 状态，而不是只发通知 |
| 节点 3：CEO 经营台 | 2-4 个月 | 把 CEO 对话、待决策、审批、项目监控、干预收口到一个工作面 | CEO 不需要在多个页面和多个入口之间来回切换 |
| 节点 4：部门操作系统 | 3-6 个月 | 让 Department 成为真正的一等经营单元 | 每个部门都有稳定的规则、技能、记忆、配额、例行任务和绩效上下文 |
| 节点 5：OKR 与记忆闭环 | 4-8 个月 | 让组织从“会执行”升级为“会学习、会复盘、会追目标” | 项目进展、部门记忆和 OKR 可以相互驱动 |
| 节点 6：自治运营与开放执行生态 | 6-18 个月 | 让公司级自治与外部执行生态成立 | CEO 能围绕目标做异常管理，底层可接多个叶子执行器 |

## 节点 1：执行底座收口

这是当前最优先的根节点。没有它，后面的“虚拟公司”都只是表面 UI。

### 目标

把现有运行时从“默认围绕某个 provider/bridge 的实现细节运转”，升级为“围绕统一执行事件契约运转”。

### 要解决的核心问题

1. `group-runtime.ts` 仍然知道太多 transport 细节。
2. `TaskExecutor` 还更像一次性任务调用，不是会话/执行 backend。
3. `Run`、`Conversation`、`Executor Handle` 之间还没有完全解耦。
4. 未来新增第三方执行后端时，系统仍然会隐式假设所有 provider 都具备一样的 watch/cancel/stream 能力。

### 关键交付

1. 定义统一的 `ExecutionBackend` 或 `AgentSessionBackend` 契约。
2. 引入统一事件流：至少覆盖 started、step、artifact、blocked、completed、failed、cancelled。
3. 引入 capability matrix：streaming、appendMessage、cancel、watch、tool-calls、artifacts、interactive。
4. 让 Stage Runtime 基于能力降级，而不是基于 provider 名字写分支。
5. 用一个最小实验 executor 验证单 stage 可跑通。

### 借鉴 craft 的正确位置

只借鉴 backend/event/capability 的抽象方式，不借 workspace/session/UI 壳。

### 完成标准

1. 同一个 stage 能在 `antigravity` 与 `codex` 路径上通过统一 lifecycle 落盘。
2. Runtime 的主逻辑不再直接依赖某个 transport 的 step 细节。
3. 新接一个实验后端时，不需要再重写 Project/Run 主链路。

### 最容易犯的错

1. 继续先接 provider，再补契约。
2. 误以为“已有 TaskExecutor”就等于执行抽象已经完成。
3. 在没有 capability 降级的情况下强推多 provider 一致行为。

---

## 节点 2：治理内核闭环

执行底座收口之后，下一步不是先做更多前端，而是把治理真正做成系统能力。

### 目标

让审批、配额、规则、人工 gate、异常干预、计划任务都不只是记录或通知，而是能够直接驱动运行时状态迁移。

### 关键交付

1. 把 approval 从“消息通知”升级为“可执行决策”。
2. 打通 quota/policy 对 dispatch 和 resume 的硬约束。
3. 统一 pause/resume/reject/retry/intervene 的状态机。
4. 建立治理审计账本，记录 CEO 为什么干预、为什么批准、为什么调额。
5. 让 scheduler 不只是触发任务，也能带上治理上下文。

### 完成标准

1. CEO 批准/拒绝能够真实改变 run/project 流程。
2. 超额、越权、缺审批等场景能阻止继续执行，而不是事后提醒。
3. 任意一次干预都能在系统中追溯理由、执行人、前后状态。

### 最容易犯的错

1. 把审批做成另一个消息中心，而不是运行时控制层。
2. 在治理主链路未打通前就做自治。
3. 把项目状态与审批状态分开维护，最后形成双真相源。

---

## 节点 3：CEO 经营台

这是第一块真正面向用户感知价值的大产品节点，但它必须建立在前两个节点之上。

### 目标

把 CEO Office 从“聊天 + 配置 + 若干分散面板”升级为“单屏经营台”。

### 经营台必须统一的能力

1. 一句话派活。
2. 待决策队列。
3. 审批队列。
4. 项目与 run 运行态监控。
5. 快速干预入口。
6. 风险和异常提醒。
7. 最近完成、最近失败、最近卡住。

### 关键交付

1. 统一 CEO chat 与项目控制台的数据面。
2. 把 `needs_decision`、approval、runs、projects 汇成一个经营视图。
3. 在同一工作面内完成：下达任务、看执行、审批、介入、复盘。
4. 给 CEO 建立会话域模型，而不是只依赖零散 conversation handle。

### 完成标准

1. CEO 不再需要在 Dashboard、Projects、Chat、Approval 之间频繁跳转。
2. 一个项目从立项到干预到复盘，可以在一个视角里闭环完成。
3. CEO 的每个操作都能落入审计账本和项目上下文。

### 最容易犯的错

1. 先做人格化 UI 和氛围感，后补经营控制能力。
2. 让 CEO 直接学会调用底层 API，却绕开治理与审计。
3. 再造一个新页面，导致已有页面继续分裂。

---

## 节点 4：部门操作系统

当 CEO 工作面成立后，下一步是把“部门”做成真正的一等经营单元。

### 目标

让每个 Department 不只是一个 workspace，而是一个拥有身份、制度、技能、记忆、预算和节奏的可运营实体。

### 部门最小经营契约

1. Department Identity：定位、职责、边界、对外接口。
2. Department Rules：规章、审批边界、风险偏好。
3. Department Skills：可调用能力库。
4. Department Memory：长期经验、近期上下文、失败教训。
5. Department Provider Policy：默认模型、例外策略、预算配额。
6. Department Routine：日报、周报、巡检、复盘、补知识。
7. Department Roster：员工/角色配置与替补机制。

### 关键交付

1. 统一 `.department/` 资产协议。
2. 让 rules、skills、memory、scheduler、quota 的 owner 都明确归属于 Department。
3. 建立部门可视化经营面：投入、产出、积压、风险、知识沉淀。
4. 让 CEO 对部门发指令时，真的能借助这些资产完成运营。

### 完成标准

1. 新建一个部门时，系统知道怎样初始化它的规章、技能、记忆和节奏。
2. 切换 provider、加技能、调规则、调配额会影响该部门后续行为。
3. 部门不是“一个目录 + 一堆手工文件”，而是有标准 contract 的经营单位。

### 最容易犯的错

1. 太早让每个部门自由生长，结果变成一堆不兼容的小系统。
2. 只有 UI 面板，没有稳定的部门资产协议。
3. 让技能、模板、记忆分散到多个目录 owner 不清。

---

## 节点 5：OKR 与记忆闭环

这是 Antigravity 从“会执行”走向“会学习、会经营”的关键节点。

### 目标

让项目推进、部门知识沉淀和目标管理形成闭环。

### 关键交付

1. 把 OKR 从静态配置升级为可计算的经营目标。
2. 让项目/阶段产出可以映射到 KR 进展。
3. 让 memory 不再只是 Markdown 归档，而是可提炼、可检索、可引用的组织知识。
4. 建立日报/周报/月报与复盘机制。
5. 让 scheduler 可以围绕目标触发 routine，而不是只定时 dispatch。

### 完成标准

1. CEO 可以看到目标偏差，而不只是项目状态。
2. 部门能基于近期失败/成功自动更新记忆与建议。
3. 例行汇报能引用真实项目证据、风险和下阶段动作。

### 最容易犯的错

1. 把 OKR 只当仪表盘字段。
2. 把 memory 只当积累文本，不做提炼与召回。
3. 把周报做成内容生成，而不是经营闭环的一部分。

---

## 节点 6：自治运营与开放执行生态

前五个节点完成后，Antigravity 才有资格进入真正的自治公司阶段。

### 目标

让 CEO 从“亲自盯执行”升级为“围绕目标做异常管理”，并允许底层接入多个叶子执行器。

### 关键交付

1. 建立经营例外机制：默认自治，异常升级给 CEO。
2. 建立跨部门协同与资源竞争治理。
3. 接入多种叶子执行后端，但统一受 Execution Backend 契约和治理层约束。
4. 允许外部 agent/backend 成为 Department 的执行供应商。
5. 建立公司级 dashboard：产能、风险、预算、目标偏差、组织学习速率。

### 完成标准

1. CEO 不必亲自盯每个项目，也能掌握经营异常。
2. 不同 executor 不会破坏 Project/Run/Stage 的真相源。
3. 系统能稳定运行多个部门、多个项目、多个周期任务。

### 最容易犯的错

1. 在前面 1-5 节点没站稳前就急着追求自治。
2. 把“多 backend”误当成“平台化完成”。
3. 没有统一治理层就开放外部执行器接入。

---

## 节点之间的依赖关系

顺序上不能反。

1. 节点 1 是地基。
2. 节点 2 是安全与治理护城河。
3. 节点 3 才是用户真正能感知的经营台。
4. 节点 4 让“部门 = 经营单元”成立。
5. 节点 5 让组织开始有学习和目标系统。
6. 节点 6 才是自治与生态开放。

如果顺序反过来，最容易得到的结果是：看起来像公司，实际上只是多个对话框、多个页面和多个 provider 的拼接物。

## 未来 2 周该先做什么

长期路线很长，但下一步不该发散。接下来两周只做三件事：

### 先回答一个关键方法论问题

如果只能在“先把 Antigravity 现有链路跑通”和“先做大抽象”之间二选一，正确答案是：

> 先打通一条最小但完整的主链路，再围绕这条主链路做抽象。

原因有三个：

1. 如果现在先做大抽象，极容易抽象出一套脱离真实运行问题的接口，最后变成新的耦合层。
2. 如果现在只顾继续堆功能、不补边界，耦合会继续固化，后面抽象代价更高。
3. 只有先跑通一条真实主链路，才能知道哪些接口是稳定边界，哪些只是当前实现细节。

所以近期策略不是“先产品”或“先抽象”的二选一，而是：

1. 先选一条最小主链路做真正确认。
2. 只围绕这条主链路抽第一层边界。
3. 抽象完成后，再扩第二条 provider/场景路径。

### 推荐的最小主链路

建议用下面这条链路做唯一优先级：

1. CEO 或标准 dispatch 创建一个项目。
2. 单 stage 被成功派发。
3. run 可以稳定进入 running、completed 或 failed。
4. artifact、review、project 状态能正确落盘。
5. CEO 可以在同一链路里看到结果并做一次干预或复盘。

只要这条链路还有断点，就不应该急着做全局 provider 抽象。

### 抽象的切入方式

当最小主链路跑通后，第一层只抽三件事：

1. Run lifecycle 事件。
2. Executor capability 声明。
3. Conversation handle 与 Project/Run 状态的边界。

不要一开始就试图把所有 provider、所有 UI 场景、所有会话模型一次抽干净。

### 任务 A：执行后端契约蓝图

输出一份底层设计稿，明确：

1. `ExecutionBackend` 接口。
2. `ExecutionEvent` 类型集合。
3. capability matrix。
4. Run/Conversation/Handle 的边界。
5. 如何让 Stage Runtime 从 transport 细节中脱身。

### 任务 B：治理状态机蓝图

输出一份治理设计稿，明确：

1. approval、quota、gate、intervene、resume 的状态迁移。
2. 哪些事件必须落审计账本。
3. 哪些写操作必须经过 CEO 受控 action service。

### 任务 C：CEO 经营台信息架构

输出一份产品蓝图，明确：

1. CEO 单屏应该有哪些区块。
2. 哪些数据是一级优先级。
3. 哪些操作要直接可点。
4. 哪些动作需要审批或二次确认。

## 不该做的事

接下来一个阶段，有三件事不建议优先做。

1. 不优先把大量精力投入更多 provider 适配。
2. 不优先迁移到底层 shell 或直接基于 craft 重做。
3. 不优先做更重的前端包装、人格包装或场景包装。

这些都会让团队感觉“功能越来越多”，但不会真正提高 Antigravity 的核心完成度。

## 你现在最担心的三件事，应该怎么排优先级

你提到的三个缺口都是真的：部门记忆、定时任务、自主演进都还没有真正做硬。
但它们的顺序不能并列。

### 第一优先级：部门记忆

先把 Department Memory 做成可读、可写、可提炼、可召回的真实能力。

原因：

1. 没有记忆，部门不会积累经验，永远只是“每次重新开始”。
2. 没有记忆，CEO 的经营台只能看状态，不能看组织学习。
3. 后续 scheduler 和自主演进都需要记忆作为输入材料。

近期目标不该是“做一个很大的知识系统”，而是先完成 Memory v1：

1. run 完成后可提炼经验。
2. 部门执行前可召回相关经验。
3. CEO 可查看部门近期新增知识和失败教训。

### 第二优先级：定时任务

Scheduler 要先做成“经营节奏系统”，而不是只做 cron 触发器。

原因：

1. 它决定公司有没有日报、周报、巡检、复盘这些固定节奏。
2. 它是 CEO 从“盯人盯项目”走向“按节奏管理”的关键。
3. 但如果前面没有 memory 和治理，scheduler 只会定时制造噪音。

近期目标建议是 Scheduler v1：

1. 定时读取项目和部门状态。
2. 自动生成日报/周报任务。
3. 触发需要 CEO 决策或审批的例外项。
4. 结果写回 memory 和审计账本。

### 第三优先级：自主演进

自主演进必须放最后，而且早期只能做“提出改进建议”，不能直接放权为“自动改系统”。

原因：

1. 自主演进本质上依赖 memory、scheduler、governance 三层都已经成立。
2. 如果没有稳定记忆和审计，它只会变成随机改配置或随机造流程。
3. 如果没有治理闭环，它很容易绕开 CEO 和审批边界。

所以近期最多只做 Self-Evolution v0：

1. 自动识别重复问题。
2. 生成改进建议或流程优化提案。
3. 提交给 CEO 审批。
4. 审批通过后由受控任务去执行，而不是直接自改。

### 一个很硬的排序结论

顺序就是：

1. 先打通最小主链路。
2. 再做 Department Memory v1。
3. 然后做 Scheduler v1。
4. 最后才碰 Self-Evolution v0。

如果顺序反过来，最容易得到的就是：

1. 没有记忆支撑的定时噪音。
2. 没有治理约束的伪自治。
3. 看起来很聪明、实际上不可控的系统行为。

## 最终判断

Antigravity 现在真正该走的，不是“把自己做得更像另一个 coding agent 工具”，而是：

> 先把执行底座和治理中枢做硬，再把 CEO、部门、记忆、OKR 和例行机制做成一个真正能长期运营项目的 AI 公司系统。

如果按这个顺序推进，Antigravity 的差异化会越来越清楚。
如果反过来推进，它会越来越像一个功能越来越多、但主线越来越散的 agent 壳。

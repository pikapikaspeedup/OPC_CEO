# 架构文档维护方法论

> 适用于任何中大型项目的架构文档与代码同步维护方法。

---

## 核心理念

**文档和代码是同一个系统的两个视图。**

- 代码描述"怎么做"
- 文档描述"做了什么、为什么、怎么用"
- 两者不同步 = 系统存在两个互相矛盾的真相

---

## 一、文档金字塔

每个项目只需要 4 层文档，自上而下：

```
             ┌──────────────┐
     L0      │ ARCHITECTURE │ ← 系统全景、模块关系、技术决策
             │    .md       │    目标读者：新加入的开发者
             ├──────────────┤
     L1      │  User Guide  │ ← 概念解释、功能说明、使用示例
             │              │    目标读者：功能使用者
             ├──────────────┤
     L2      │ API Reference│ ← 端点、参数、响应格式
             │              │    目标读者：集成方 / 自动化脚本
             ├──────────────┤
     L3      │  CHANGELOG   │ ← 变更记录
             │              │    目标读者：升级者
             └──────────────┘
```

### 各层定位

| 层级 | 核心问题 | 更新频率 | 典型篇幅 |
|:-----|:---------|:---------|:---------|
| L0 架构 | 系统由哪些模块组成？它们如何协作？ | 架构变更时 | 500-2000 行 |
| L1 用户指南 | 这个功能是什么？怎么用？有什么限制？ | 功能变更时 | 每功能 50-100 行 |
| L2 API 参考 | 这个端点接收什么参数？返回什么？ | 接口变更时 | 每端点 20-40 行 |
| L3 变更记录 | 这次改了什么？ | 每次任务完成 | 每次 10-30 行 |

## 二、架构文档（L0）的标准结构

```markdown
# {项目名} — 产品架构

## 系统全景
<!-- Mermaid 全局架构图：所有子系统 + 外部依赖 + 数据流方向 -->

## 模块依赖关系
<!-- Mermaid 依赖关系图：分层展示模块间依赖 -->

## §1. {子系统A}
### 数据流 / 状态机
<!-- Mermaid sequence / state diagram -->
### 关键文件
<!-- 模块文件 → 职责 对应表 -->

## §2. {子系统B}
...

## API 端点总览
<!-- 全部 API 路由的一览表 -->

## 技术栈
## 目录结构
```

### 关键原则

1. **一个子系统一个章节** — 章节数 ≈ `src/lib/` 或 `src/modules/` 下的目录数
2. **每章必须有"关键文件"表** — 列出该子系统所有 > 100 行的模块文件及其职责
3. **用 Mermaid 可视化** — 架构图比文字描述更能传递结构信息
4. **API 总览表是全局索引** — 路由数 ≈ 代码中的 route 文件数

## 三、文档 vs 代码差异检测

### 原理

用代码的物理结构（文件数、目录数、路由数）作为"客观真相"，文档作为"主观描述"，对比两者找差异。

### 通用检测方法

| 检测项 | "真相"来源 | 文档来源 | 检测方式 |
|:-------|:----------|:---------|:---------|
| API 路由覆盖 | `find src/app/api -name route.ts` | ARCHITECTURE API 表 | 行数对比 |
| 模块文件覆盖 | `ls src/lib/{module}/*.ts` | 关键文件表 | comm 差集 |
| 子系统覆盖 | `ls -d src/lib/*/` | ARCHITECTURE 章节标题 | 目视对比 |
| 数据模型字段 | `interface` 定义 | 用户指南字段表 | grep 关键类型 |

### 检测脚本模板

```bash
#!/bin/bash
# doc-health-check.sh — 文档健康检查

echo "=== API Coverage ==="
code_routes=$(find src -path "*/api/*" -name "*.ts" -o -name "route.*" | wc -l)
doc_routes=$(grep -c '/api/' ARCHITECTURE.md 2>/dev/null || echo 0)
echo "Code routes: $code_routes | Doc routes: $doc_routes"
if [ "$((code_routes - doc_routes))" -gt 3 ]; then
  echo "⚠️  Gap > 3, review needed"
fi

echo ""
echo "=== Module Coverage ==="
for dir in src/lib/*/; do
  module=$(basename "$dir")
  count=$(ls "$dir"*.ts 2>/dev/null | grep -v test | wc -l)
  doc_count=$(grep -c "$module" ARCHITECTURE.md 2>/dev/null || echo 0)
  if [ "$count" -gt 3 ] && [ "$doc_count" -lt 2 ]; then
    echo "⚠️  $module: $count files but only $doc_count doc mentions"
  fi
done

echo ""
echo "=== Section Count vs Dir Count ==="
sections=$(grep -c '^## [0-9]' ARCHITECTURE.md 2>/dev/null || echo 0)
dirs=$(ls -d src/lib/*/ 2>/dev/null | wc -l)
echo "ARCHITECTURE sections: $sections | src/lib dirs: $dirs"
```

## 四、文档同步触发规则

### 变更类型 → 文档动作映射

| 代码变更 | L0 架构 | L1 用户指南 | L2 API 参考 | L3 变更记录 |
|:---------|:--------|:-----------|:-----------|:-----------|
| 新增 API 端点 | API 表 +1 行 | 如有新概念则新增节 | 新增端点文档 | ✅ |
| 新增子系统目录 | **新增章节** + 依赖图更新 | **新增功能节** | — | ✅ |
| 重构/拆分模块 | 关键文件表更新 | — | — | ✅ |
| 修改数据模型 | — | 字段表更新 | 请求/响应更新 | ✅ |
| 修改执行流程 | 状态机/序列图更新 | 流程说明更新 | — | ✅ |
| 新增配置项 | — | 配置示例更新 | — | ✅ |
| Bug 修复 | — | — | — | ✅ |

### 快速判断流程

```
代码有改动？
├── 改了 route/API？ ──→ 更新 API 参考 + 架构 API 表
├── 新建了目录？ ──→ 架构新章节 + 用户指南新节
├── 重构了模块？ ──→ 架构关键文件表
├── 改了类型定义？ ──→ 检查用户指南字段表
└── 所有情况 ──→ 更新变更记录
```

## 五、防退化检查清单

在以下时间点执行检查：

### 每次任务完成后（2 分钟）

- [ ] 变更记录已更新
- [ ] 新增的 route.ts 在 API 表中有对应行
- [ ] 新增的模块文件在关键文件表中

### 每周（15 分钟）

- [ ] 运行文档健康检查脚本
- [ ] API 路由数差异 < 3
- [ ] 无新增子系统目录缺少 ARCHITECTURE 章节

### 每次重大重构后（30 分钟）

- [ ] 模块依赖关系图与代码一致
- [ ] 关键文件表覆盖所有 > 100 行的模块
- [ ] 用户指南中的概念/术语与代码命名一致

## 六、Mermaid 图的使用策略

| 图类型 | 适用场景 | 示例 |
|:-------|:---------|:-----|
| `graph TB/LR` | 系统架构、模块依赖 | 全景图、分层依赖图 |
| `sequenceDiagram` | 时序交互（API 调用链） | 请求流、审批流 |
| `stateDiagram` | 状态机（对象生命周期） | Run 状态、Pipeline 状态 |
| `erDiagram` | 数据模型关系 | 项目-Run-Pipeline 关系 |
| `flowchart` | 流程决策 | Provider 解析、CEO 命令路由 |
| `mindmap` | 功能树 | CLI 命令总览 |

### 原则

1. **一个图只表达一个概念** — 不要在一个图里塞 20 个节点
2. **用 subgraph 分组** — 按逻辑层/子系统分组
3. **边上标注协议/数据** — `-->|"gRPC"| Server` 比裸箭头信息量大
4. **保持更新** — 过时的图比没有图更糟糕

## 七、反模式（应避免的做法）

| 反模式 | 问题 | 正确做法 |
|:-------|:-----|:---------|
| 代码注释代替文档 | 分散、不全面、无法导航 | 注释说"为什么"，文档说"是什么和怎么用" |
| 文档写完就不管了 | 3 个月后全部过时 | 用检测脚本定期审查 |
| 一份文档覆盖所有 | 几千行无法阅读 | 按金字塔分层，每层独立 |
| 只写"怎么做"不写"做了什么" | 新人看不懂代码的意图 | 架构图 + 概念解释优先 |
| API 文档手写 + 不同步 | 和代码不一致 | 物理指标对比（路由数 vs 文档行数） |
| 重构后不更新依赖图 | 图和代码矛盾 | 重构完成 → 重画图 |

## 八、适应不同项目规模

| 项目规模 | L0 架构 | L1 指南 | L2 API | L3 变更 |
|:---------|:--------|:--------|:-------|:--------|
| 小型（< 20 文件） | README 内一节即可 | README 内 | 不独立写 | Git commit log |
| 中型（20-100 文件） | 独立文件，3-5 章 | 1 份指南 | 1 份 API 参考 | CHANGELOG.md |
| 大型（> 100 文件） | 独立文件，10+ 章 | 多份主题指南 | 分端点文档 | 结构化进度文档 |

---

## 总结：三句话

1. **文档的信息来源是代码**，用代码的物理结构验证文档的完整性
2. **变更触发更新**，不要等到"有空了再写"
3. **自动检测优于人工记忆**，脚本跑一次 2 秒，人想一次 20 分钟
